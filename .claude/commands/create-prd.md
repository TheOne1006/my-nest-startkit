# 创建 PRD: 生成产品需求文档

## 概述

基于当前的对话上下文和讨论的需求，生成一份全面的产品需求文档 (PRD)。使用下面定义的结构和章节来创建一份详尽、专业的 PRD。

## 输出文件

将 PRD 写入: `$ARGUMENTS` (默认: `PRD.md`)

## PRD 结构

创建一个结构良好的 PRD，包含以下章节。根据可用信息调整深度和细节：

### 必需章节

**1. 执行摘要**
- 简明的产品概述 (2-3 段)
- 核心价值主张
- MVP 目标声明

**2. 使命**
- 产品使命声明
- 核心原则 (3-5 个关键原则)

**3. 目标用户**
- 主要用户画像
- 技术舒适度
- 关键用户需求和痛点

**4. MVP 范围**
- **范围内:** MVP 的核心功能 (使用 ✅ 复选框)
- **范围外:** 推迟到未来阶段的功能 (使用 ❌ 复选框)
- 按类别分组 (核心功能, 技术, 集成, 部署)

**5. 用户故事**
- 主要用户故事 (5-8 个故事)，格式为："作为 [用户]，我想要 [行动]，以便 [好处]"
- 为每个故事包含具体的例子
- 如果相关，添加技术用户故事

**6. 核心架构与模式**
- 高级架构方法 (NestJS 模块化架构)
- 目录结构 (如果适用)
- 关键设计模式和原则 (DI, Repository, DTO)
- 特定技术模式 (Prisma, Guards, Interceptors)

**7. 工具/功能**
- 详细的功能规格
- 如果构建代理：工具设计及其用途、操作和关键功能
- 如果构建应用：核心功能细分

**8. 技术栈**
- 后端/前端技术及其版本 (NestJS, TypeScript, Prisma, PostgreSQL)
- 依赖项和库
- 可选依赖项
- 第三方集成

**9. 安全与配置**
- 认证/授权方法 (JWT, RBAC)
- 配置管理 (环境变量, `config/` 模块)
- 安全范围 (范围内和范围外)
- 部署注意事项 (Docker)

**10. API 规范** (如果适用)
- 端点定义 (Swagger/OpenAPI)
- 请求/响应格式
- 认证要求
- 示例负载

**11. 成功标准**
- MVP 成功定义
- 功能需求 (使用 ✅ 复选框)
- 质量指标 (测试覆盖率, 性能)
- 用户体验目标

**12. 实施阶段**
- 分解为 3-4 个阶段
- 每个阶段包括：目标, 交付物 (✅ 复选框), 验证标准
- 现实的时间表估计

**13. 未来考虑**
- MVP 后增强
- 集成机会
- 后期阶段的高级功能

**14. 风险与缓解**
- 3-5 个关键风险及其具体的缓解策略

**15. 附录** (如果适用)
- 相关文档
- 关键依赖项及其链接
- 存储库/项目结构

## 说明

- 确保内容针对 NestJS Startkit 技术栈进行了定制。
- 使用清晰、专业的语言。
- 保持格式一致且易于阅读。
